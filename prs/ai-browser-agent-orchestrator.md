# AI 浏览器代理调度器 — 智能网页任务编排系统

> **一句话卖点**：运营人员每天花 2 小时在多个网站间切换做重复操作，用调度器定义一次任务 DSL，10 个 Agent 并行执行，从 2 小时变成 5 分钟

## 概述

基于 Issue #153 创建。解决多 AI 浏览器 Agent 协调痛点——现有方案（browser-use、Playwright）是单 Agent，多任务场景需要手动串联。本系统提供：

- **任务 DSL**：用 YAML/JSON 定义多步骤网页任务
- **智能调度器**：根据任务类型自动分配给最合适的 Agent
- **资源隔离**：每个 Agent 独立 browser context，避免状态冲突
- **优先级队列**：紧急任务插队，不阻塞正常流程

## 竞品分析

### 现有方案对比

| 维度 | Zapier/n8n/Make | UiPath/影刀 (RPA) | 本方案 |
|------|-----------------|-------------------|--------|
| **核心定位** | API 集成自动化 | 桌面级操作自动化 | AI 驱动的网页任务编排 |
| **技术门槛** | 低（可视化拖拽） | 高（需要专业培训） | 中（DSL + 可视化） |
| **网页交互能力** | 弱（依赖 API） | 强（像素级操作） | 强（AI 理解 + DOM 操作） |
| **多 Agent 协调** | 无（单线程执行） | 有限（需要额外编排） | 原生支持 |
| **智能容错** | 无（规则驱动） | 弱（需要预设规则） | 强（AI 自动适应） |
| **成本** | $19-599/月 | $420+/月（企业定价） | ¥99-999/月 |

### 为什么用户会选择本方案？

1. **vs Zapier/n8n**：
   - 目标网站没有 API？调度器可以直接操作网页
   - 需要视觉理解（截图、验证码）？AI Agent 原生支持
   - 多任务并行？10 个 Agent 同时跑

2. **vs RPA 工具**：
   - 不需要安装客户端，纯云端执行
   - AI 自适应页面变化，不需要维护脆弱的选择器
   - 价格更低，中小企业可负担

3. **vs 纯 Playwright/browser-use**：
   - 不需要写代码，DSL + 可视化编辑器
   - 多 Agent 自动调度，不用手动编排
   - 内置监控、重试、断点续传

## 典型用户场景

### 场景 1：电商运营 - 竞品价格监控
- **用户**：小美，某服装品牌电商运营，负责 5 个店铺
- **痛点**：每天要检查 20+ 竞品店铺的价格变动，手动记录到 Excel，耗时 2 小时
- **使用方式**：
  1. 定义任务 DSL：访问竞品店铺 → 搜索指定商品 → 截图价格 → 记录到 Airtable
  2. 设置每天早上 8 点自动执行
  3. 10 个 Agent 并行，5 分钟完成
- **效果**：每天节省 2 小时，价格变动第一时间发现

### 场景 2：内容运营 - 多平台内容同步
- **用户**：阿杰，自媒体博主，在公众号、知乎、小红书、B 站发内容
- **痛点**：一篇内容要手动登录 4 个平台发布，排版要分别调整
- **使用方式**：
  1. 定义任务 DSL：读取 Notion 内容 → 登录各平台 → 智能适配格式 → 发布
  2. 一键触发，4 个 Agent 同时执行
- **效果**：发布时间从 30 分钟变成 3 分钟

### 场景 3：数据分析师 - 定期数据采集
- **用户**：老王，投资机构分析师，需要每周采集 50 家公司的公开数据
- **痛点**：数据分散在各网站（财报、新闻、社交媒体），手动采集容易遗漏
- **使用方式**：
  1. 定义任务 DSL：访问指定网站 → 提取关键数据 → 聚合到 Google Sheets
  2. 设置每周一自动执行，异常自动告警
- **效果**：采集时间从 1 天变成 10 分钟，零遗漏

## 功能设计

### 核心功能

1. **任务定义层**
   - 可视化任务编辑器（拖拽式）
   - YAML/JSON DSL 语法
   - 支持条件分支、循环、变量传递
   - 任务模板市场（社区分享）

2. **智能调度层**
   - 任务类型识别（数据采集 vs 自动化操作 vs 监控）
   - Agent 能力匹配（哪个 Agent 擅长处理这类任务？）
   - 负载均衡（避免某个 Agent 过载）
   - 失败重试 + 备选 Agent 切换

3. **执行监控层**
   - 实时任务状态仪表盘
   - 异常检测（页面结构变化、反爬触发）
   - 断点续传（长任务失败后从断点恢复）
   - 执行日志 + 回放

4. **协作层**
   - Agent 间上下文共享（MCP 协议）
   - 结果聚合（多个 Agent 的输出合并）
   - 冲突检测（两个 Agent 同时操作同一网站）

## 技术方案

### 数据层

- **任务存储**：PostgreSQL + JSONB（任务 DSL）
- **执行状态**：Redis（实时状态）+ TimescaleDB（历史记录）
- **会话管理**：Redis（browser context 持久化）
- **任务队列**：Redis Streams（优先级队列）

### 逻辑层

```
任务提交 → DSL 解析 → 任务分析 → Agent 匹配 → 资源分配 → 执行监控 → 结果聚合
```

**核心组件**：

1. **DSL Parser**：YAML/JSON → 内部任务图
2. **Task Analyzer**：识别任务类型、估算资源需求
3. **Agent Matcher**：基于历史表现匹配最佳 Agent
4. **Resource Manager**：管理浏览器实例池
5. **Execution Engine**：Playwright + browser-use 集成
6. **Monitor**：实时监控 + 异常检测

### 展示层

- **Web UI**：React + TypeScript
- **任务编辑器**：Monaco Editor（代码模式）+ React Flow（可视化模式）
- **监控仪表盘**：Grafana 或自研
- **CLI**：支持脚本化调用

|------|------|----------|----------|--------|
| 免费版 | ¥0 | 50 任务/月 | ¥6 | 引流亏本 |

### 成本优化策略
1. **浏览器实例复用**：空闲实例不销毁，热启动节省 80% 时间
2. **智能调度**：相似任务合并执行，减少重复页面加载
3. **代理池分级**：普通网站用便宜代理，反爬严格网站用高质量代理
4. **AI 按需调用**：简单任务不用 AI，复杂任务才启用

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 反爬对抗 | Agent 被封禁 | 代理池 + 请求频率限制 + User-Agent 轮换 |
| 网页结构变化 | 任务失效 | AI 视觉备选（GPT-4V 定位元素）+ 社区维护选择器库 |
| 浏览器实例成本高 | 利润率低 | 智能回收机制 + 按需启动 + 容器化隔离 |
| Agent 间竞态条件 | 数据冲突 | 分布式锁 + 乐观并发控制 + 冲突检测 |
| 任务 DSL 学习曲线 | 用户流失 | 可视化编辑器 + 自然语言生成 DSL |
| 任务市场合规风险 | 监管问题 | 初期仅支持免费分享，商业化需取得相关资质 |
| 用户数据隐私 | 信任危机 | 端到端加密 + 本地存储选项 + 透明隐私政策 |
